# 自動車追い越し検出 (GoPro)

自転車に取り付けた GoPro 動画(前向き / 後ろ向き)から、
**自動車に追い越される瞬間**を自動検出し、区間を CSV 出力・クリップ切り出しするツール。
研究用途(追い越し時間の把握、離隔距離分析の前段)を想定。

撮影方向は `--view {front,rear}` で指定する(必須)。日本(左側通行)では追い越し車は
自転車の右側を抜けていくため、見え方が次のように異なる:

- **`--view front`(前向き)**: 追い越し車は画面の**右端付近から現れ、左(中央)へ流れて**抜ける。
- **`--view rear`(後ろ向き)**: 後方カメラは左右反転するため、追い越し車は**画面中央(消失点)付近に
  小さく現れ、左端へ大きくなりながら流れて**抜ける(前方とは奥行きの進行が逆)。

前方では路駐車両などが画面中央(消失点)付近から現れて外側の端へ流れ、後方では対向車が
逆に端付近から中央へ遠ざかる。上記の「出現位置 + 流れる向き」の条件でこれらを除外し、
追い越しの誤検知を減らせる。

## セットアップ

このプロジェクトは GitHub には軽量なソースだけを置き、初回起動時に必要な実行環境を
PC上へ自動作成する。利用者が実行するファイルは `起動.bat` のみ。

```powershell
# 初回起動。portable\python、YOLOモデル、FFmpeg が無ければ自動準備する
.\起動.bat
```

初回セットアップでは次を自動で準備する。

- `portable\python` ... Python 本体と `requirements.txt` のライブラリ
- PyTorch ... CUDA対応GPUを検出した場合はGPU版、検出できない場合はCPU版を自動インストール
- `yolov8n.pt` ... YOLO の既定モデル
- `tools\ffmpeg.exe` / `tools\ffprobe.exe` ... クリップ切り出し用。既に PATH 上に ffmpeg がある場合は省略可

セットアップにはインターネット接続が必要。途中で失敗した場合、次回 `起動.bat` 実行時に
壊れた `portable\python` を検出して再作成する。

GitHub から取得した作業フォルダでは、起動時に更新確認も行う。ローカル変更が無い場合だけ
`git pull --ff-only` で自動更新し、作業中の変更がある場合は自動更新をスキップする。

## 使い方

```powershell
# GUIを起動
.\起動.bat

# 起動環境だけ確認
.\起動.bat --check

# 更新確認を省略してGUIを起動

# コマンドラインで動画検出
.\起動.bat 入力動画.mp4 rear

# オプションを追加して検出

```

- ダブルクリックして GUI を起動する

- `起動.bat --check` でPython・CUDA環境だけを確認する
- `起動.bat --no-update` で更新確認を省略して起動する

## GUIワークフロー

非エンジニア向けの一連の作業画面は `起動.bat` をダブルクリックして起動する。
画面は HTML/CSS/JavaScript で作られ、Windows WebView2上にデスクトップアプリとして表示される。

1. **動画検出**: GoPro動画とカメラ方向を選び、追い越し候補とクリップを作成する
2. **クリップ確認**: 候補クリップを開き、正しい追い越しは「採用」、誤検知は「除外」にする
3. **PCAP・時刻同期**: Velodyne PCAPを読み込み、GPS自動同期でGoProとVeloViewの時刻差(オフセット)を求める
4. **点群・出力**: 候補時刻へ移動して車両点群を矩形選択し、フレーム別CSVへ出力する。
   「レビュー結果をCSV保存」で確認結果を保存する

**除外**にした候補は、手順3の時刻一覧と手順4の候補選択の対象から自動的に除かれる。

LiDAR画面はclassic PCAP内の標準1206-byte Velodyneデータパケットを直接読み込む。
VLP-16、HDL-32E、VLP-32Cに対応し、点群は上面図で表示される。ドラッグで対象車両を囲い、
開始・終了フレームを指定して出力すると、各フレームの選択点が個別CSVになる。
CSVの9・10列目は従来工程と同じ `Points:0`（X）・`Points:1`（Y）。

GUIの入力内容とショートカット設定は `gui_settings.json` に保存される。確認済みイベントは
`out/reviewed_overtaking_events.csv` に保存され、次回起動時にも引き継がれる。
確認用クリップはGUI内の動画プレイヤーで再生できる。

候補確認画面のショートカット（既定値）:

- `A`: 採用
- `D`: 除外
- `W`: 未確認へ戻す
- `B` / `N` / `M`: 選択中候補の車種を 軽 / 普通 / 大型 に変更
- `Space`: 再生 / 一時停止
- `←` / `→`: 前後の候補へ移動

採用・除外のキーと車種キーは候補確認画面の「⌨ キー設定」から変更でき、`gui_settings.json` に
保存される。採用または除外を決定すると、次の候補へ自動的に移動して再生する。候補一覧の開始・
終了時刻は `◯分◯秒` 表記で表示される。車種は **大型・普通・軽** の3分類で表示され、
候補一覧の選択欄から手動修正できる。
旧形式のCSVには黄色ナンバー情報が保存されていないため、軽判定を反映するには動画検出を
再実行する必要がある。旧結果を読み込んだ場合はGUIに再解析の警告が表示される。
再解析時は新しい自動判定を採用し、候補一覧でユーザーが手動変更した車種だけを引き継ぐ。

GUIを開かず起動環境だけ確認する場合:

```powershell
.\起動.bat --check
```

## 主なオプション

| オプション | 既定 | 説明 |
|-----------|------|------|
| `--view` | (必須) | `front` / `rear`。撮影方向。追い越し車の出現位置と流れる向きを決める |
| `--out` | `out` | 出力先ディレクトリ |
| `--model` | `yolov8n.pt` | YOLO モデル。`n`<`s`<`m`<`l` で精度↑速度↓ |
| `--conf` | `0.30` | 検出信頼度しきい値 |
| `--appear-edge` | `0.70` | 出現端しきい値。front は cx≥これ(右端側), rear は cx≤1−これ(左端側)で出現する車のみ対象。大きいほど「本当に画面端から入った車」に限定 |
| `--exit-edge` | `0.40` | 反対側へ進んで消える車(横切り)を除外。front は cx≤これで消える車, rear は cx≥1−これで消える車を除外。0.40で「中央より左/右へ進んだ車」を除外 |
| `--no-stop-filter` | off | 自車運動ゲートを無効化(既定は有効) |
| `--stop-flow` | `0.8` | 自車が「動いている」と見なす背景フロー量(縮小320px基準のpx中央値)。track のフロー中央値がこれ未満なら停止中として除外。CSVの `ego_motion` を見て調整 |
| `--min-track-seconds` | `0.6` | 最低追跡時間。短時間だけ映る横切り・不安定trackを除外 |
| `--min-vertical-motion` | `0.025` | 道路奥行き方向に必要なbbox下端の移動量(画面高比) |
| `--max-lateral-vertical-ratio` | `8.0` | 横移動÷縦移動の最大値。横方向主体の交差車両を除外 |
| `--min-area-ratio` | `1.15` | 接近・離脱として必要なbbox面積の変化倍率 |
| `--max-mid-peak-ratio` | `1.8` | 中盤だけbboxが大きくなる横切り軌跡の除外倍率 |
| `--no-trajectory-filter` | off | 2次元軌跡フィルタを無効化 |
| `--no-area-filter` | off | bbox面積変化フィルタを無効化 |
| `--road-roi` | off | 台形の道路ROI滞在率フィルタを有効化。カメラごとの調整が必要 |
| `--kei-plate-threshold` | `0.45` | 黄色ナンバー形状の判定スコア。上げるほど軽判定を厳しくする |
| `--kei-min-frames` | `2` | 軽判定に必要な黄色ナンバー検出フレーム数 |
| `--kei-large-override-threshold` | `0.75` | truck/busの強判定を軽へ覆す場合に必要な黄色スコア |
| `--large-vote-threshold` | `0.70` | 大型判定に必要なtruck/bus投票率。画面端の票は弱く補正される |
| `--large-min-frames` | `5` | 大型判定に必要なtruck/bus検出フレーム数 |
| `--margin` | `2.0` | クリップ前後マージン(秒) |
| `--clip` | off | ffmpeg でクリップ切り出し |
| `--overlay` | off | 検出枠つき可視化動画を出力 |
| `--stride` | `1` | N フレームに 1 回処理(高速化、ただし追跡が途切れやすい) |
| `--device` | `auto` | 推論デバイス。`auto`=GPUがあれば自動 / `0`=GPU / `cpu` |
| `--imgsz` | `640` | 推論解像度。GUIも車種精度を優先して640を使用。下げると高速・低精度(例 416, 320) |
| `--half` | off | FP16半精度推論。GPU使用時は自動で有効 |
| `--batch` | `32` | 検出をまとめるフレーム数。大きいほど高速だがVRAM増。4KでOOMなら下げる |
| `--prefetch-batches` | `2` | RAMへ先読みするバッチ数。動画読込とGPU推論を並列化 |
| `--decoder` | `auto` | `auto` / `opencv` / `nvdec`。autoはGPU推論時にNVDECを試し、失敗時はOpenCVへ戻る |
| `--flow-stride` | `3` | 背景移動計算の間隔。Nフレーム間の移動量を使い、中間は直近値を再利用 |

## 速度・高速化

検出を**バッチ推論**し、追跡(ByteTrack)だけ逐次更新する構成。1フレームずつ
処理するより GPU 使用率が上がり、実測で数倍高速(640px・RTX2070で約8倍)。

優先度の高い高速化:

1. **GPU版 torch**(最重要)。CPU版だと桁違いに遅い。初回セットアップでは CUDA対応GPUを検出した場合にGPU版、検出できない場合にCPU版を自動インストールする。導入確認:
   ```powershell
   .\起動.bat --check
   ```
   CUDA対応GPUがあるのに `CUDA: False` になる場合は、NVIDIAドライバを確認する。PyTorch の取得先を手動指定したい場合だけ、初回セットアップ前に次の環境変数を指定してから `起動.bat` を実行する:
   ```powershell
   $env:TORCH_INDEX_URL = "https://download.pytorch.org/whl/cu128"
   .\起動.bat
   ```
2. **`--imgsz 416`(または 320)** で推論解像度を下げる(精度とのトレードオフ)。
3. **`--batch`** を VRAM が許す範囲で大きく(既定32)。OOM 時は下げる。
4. **`--overlay` を付けない**(可視化動画の書き出しは I/O コストが高い)。
5. 既定で動画をRAMへ2バッチ先読みし、読込とGPU推論を並列実行する。メモリを
   節約したい場合は `--prefetch-batches 0`、余裕があれば2～4を目安にする。
6. 背景フローは既定で3フレームごとに計算する。`--flow-stride 1`で従来どおり
   毎フレーム計算できるが、通常は3のほうが高速で検出結果も維持しやすい。
7. **GPU動画読込(NVDEC)** は既定の `--decoder auto` で自動判定される。ただしBGR画像をCPUへ
   戻す構成のため、PCや動画によってはOpenCVより遅い。本環境の実測ではOpenCVを既定とした。
8. `--stride` はフレーム間引きで速くなるが、追跡が途切れて誤検知・取りこぼしが
   増えるため非推奨。基本は上記で稼ぐ。

## 出力

- `out/overtaking_events.csv` — イベント一覧
  - `t_start_s / peak_t_s / t_end_s` … 出現 / 最接近 / 消失の時刻
  - `side` … 追い越し側 (left/right)
  - `peak_bbox_area` … 最接近時の bbox 面積(画面比、情報用)
  - `area_change_ratio` … 期待する接近・離脱方向への面積変化倍率
  - `vertical_motion` … 期待する道路奥行き方向への移動量(画面高比)
  - `lateral_vertical_ratio` … 横移動÷縦移動。大きいほど交差交通の疑いが強い
  - `road_ratio` … bbox下端中央が道路ROI内にあった割合
  - `class` … 大型・普通・軽の3分類
  - `detector_class` … YOLOの元分類(car/truck/bus)
  - `kei_plate_score / kei_plate_hits` … 黄色ナンバー判定のスコアと検出フレーム数
  - `ego_motion` … track 区間の自車背景フロー中央値(px)。`--stop-flow` 調整の目安
  - `proximity_hint` … near/mid/far の目安
- `out/clips/event_NNN_side.mp4` — 切り出しクリップ(`--clip` 時)
- `out/annotated.mp4` — 可視化動画(`--overlay` 時)

## 検出ロジック

次の条件をすべて満たす track を「追い越し」とする（主に横位置 cx で判定）:

1. **車両検出**: 各フレームを YOLO で四輪車両を検出 (car/truck/bus)
2. **5 フレーム以上追跡**: ByteTrack で同一車両に ID 付与し、5 サンプル以上
3. **追い越し側から抜けている**: front は右端側(cx ≥ `--appear-edge`)で出現 /
   rear は左端側(cx ≤ 1−`--appear-edge`)で最接近して抜ける(消失点＝中央から現れる)
4. **追い越し方向へ移動**: front は右端→中央へ左流れ / rear は中央→左端へ左流れ
   （後方は左右反転のため、前方と逆に中央から端へ向かう）
5. **逆向きに横切る車は除外**: front は左端(cx ≤ `--exit-edge`)で消える車 /
   rear は右端(cx ≥ 1−`--exit-edge`)から現れる車（＝横切り・遠ざかる対向車）を除外
6. **自車運動ゲート**(既定有効): 自転車が**停止中**(背景がほぼ動かない)に
   発生した track を除外。信号待ち・T字路など、停止中に横を通る/陰から現れる車の
   誤検知を抑える。連続フレーム間の背景オプティカルフロー(縮小320px)の中央値を
   自車速度の代理とし、track のフロー中央値が `--stop-flow` 未満なら停止と判定。
   `--no-stop-filter` で無効化。
7. **2次元軌跡フィルタ**: bbox下端中央の縦移動を道路奥行き方向の動きとして扱う。
   横移動だけが大きいtrackを、交差道路を横切る車両として除外する。
8. **面積変化フィルタ**: 後方カメラでは接近に伴う拡大、前方カメラでは離脱に伴う
   縮小を必須とする。変化が逆の車両と、中盤だけ大きくなる横切り車両を除外する。
9. **道路ROI**(任意): `--road-roi` 指定時、bbox下端中央が台形の道路領域内に
   一定割合以上存在することを必須にする。固定カメラ向けで、消失点と道路境界は
   `--vanishing-x`、`--vanishing-y`、`--road-bottom-left`、
   `--road-bottom-right` で調整する。
10. **車種分類**: YOLOのtruck/busは「大型」、carは「普通」とする。追跡区間全体を調べ、
    carの車体下部中央に黄色いナンバー形状が連続フレームで確認できた場合だけ「軽」とする。
    最接近時に側面しか映らない場合も、通過後にナンバーが見えるフレームを利用する。
    逆光・ぼけなどでナンバーが見えない場合に備え、GUIで3分類を手動修正できる。
    GoProの広角歪みで画面端の車両が大きく見えるため、車種投票にはbbox面積を使わず、
    画面端の票を弱くする。truck/bus票が70%以上で複数フレーム続く場合だけ「大型」とし、
    car票が混在するバンは「普通」側へ判定する。大型車の黄色い注意表示で「軽」にならない
    よう、強いtruck/bus判定を覆す場合は黄色ナンバースコア0.75以上を要求する。

各フィルタの診断値はCSVにも出力される。実動画で取りこぼしが増えた場合は、まず
`--min-area-ratio` と `--min-vertical-motion` を小さくして調整する。道路ROIは
GoProの向き・画角で変わるため既定では無効になっている。

> **`--stop-flow` の調整**: 既定 0.8 は撮影の速度・fps・解像度で最適値が変わる。
> まず一度走らせ、CSV の `ego_motion` 列で「走行中の追い越し」と「停止中の誤検知」の
> 値域を見比べ、その中間にしきい値を設定するとよい。
>
> **注意**: 背景フローは少数の移動物体には中央値で頑健だが、追い越し車が画面の
> 大部分を占めるほど近い場合は値が膨らみ、停止中でも走行と誤判定しうる。

## 既知の制約 / 今後

- **離隔距離は未対応**: `peak_bbox_area` は「近さの相対指標」であり実距離(m)ではない。
  単眼カメラから実距離を出すにはカメラ校正＋車幅などの既知サイズが必要。
  正確な離隔距離は超音波/レーダー等のセンサ併用が確実。
- **車線別**: 現状は「left/right(自分から見た側)」のみ。厳密な車線番号には
  路面の車線境界線(レーンマーカー)検出の追加が必要。
- GoPro GPS/IMU(GPMF) テレメトリは自車速度・進路の付加情報として今後連携可能
  (追い越し継続時間や接近相対速度の推定に有用)。
