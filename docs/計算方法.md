# データ算出方法資料

## 1. 算出フロー

本ツールのデータ算出は、次の順で行う。

1. GoPro 動画から追い越し候補を検出する。
2. 候補を目視確認し、採用・除外・車種を確定する。
3. GoPro と LiDAR の時刻オフセットを求める。
4. 採用候補の時刻に対応する LiDAR フレームを開く。
5. 対象車両の点群を矩形選択し、フレーム別に保存する。
6. 各フレームの最近接点から軌跡 CSV を作る。
7. 軌跡からフェーズ、離隔距離、通過時間、相対速度、TTC を算出する。
8. GoPro GPS 速度と統合し、自転車速度・自動車速度を最終 Excel に出力する。

主な実装ファイルは次のとおり。

| 処理 | 実装 |
|---|---|
| 追い越し候補検出 | `detect_overtaking.py` |
| GUI とレビュー保存 | `web_gui.py` |
| GoPro GPS 抽出 | `extract_telemetry.py` |
| GoPro/LiDAR 時刻同期 | `gpmf_sync.py` |
| LiDAR PCAP 読み込み・点群出力 | `lidar_pcap.py` |
| 最近接点 CSV と分析値算出 | `edit_csv.py` |
| 最終 Excel 出力 | `final_export.py` |

## 2. 追い越し候補の検出方法

### 2.1 車両検出と追跡

各フレームに対して YOLO を実行し、`car`、`truck`、`bus` を四輪車両として検出する。検出結果は ByteTrack で同一車両ごとの track にまとめる。

候補にするには、track が次の最低条件を満たす必要がある。

- 5 サンプル以上ある。
- track 継続時間が `--min-track-seconds` 以上である。
- YOLO 検出と追跡が同一車両として維持されている。

### 2.2 位置と移動方向

画面内の車両位置は bbox の中心座標 `cx` と、bbox 下端中央の `bottom_y` を使う。`cx` は画面幅で正規化された 0 から 1 の値である。

track の先頭 1/4 区間と末尾 1/4 区間について中央値を取り、次を算出する。

- `cx_start`: 出現側の代表横位置
- `cx_end`: 消失側の代表横位置
- `y_start`: 出現側の代表 bbox 下端位置
- `y_end`: 消失側の代表 bbox 下端位置
- `area_start`: 出現側の代表 bbox 面積
- `area_end`: 消失側の代表 bbox 面積

撮影方向ごとの追い越し条件は次のとおり。

| 撮影方向 | 追い越し車の想定挙動 | 主な条件 |
|---|---|---|
| 前方 `front` | 画面右端から現れ、中央側へ流れる | `cx_start >= appear_edge` かつ `cx_end < cx_start` |
| 後方 `rear` | 中央付近から現れ、左端へ大きくなりながら流れる | `cx_end <= 1 - appear_edge` かつ `cx_end < cx_start` |

横切り車両や対向車のような逆方向の動きは、`exit_edge` によって除外する。

### 2.3 自車運動ゲート

停止中に横を通る車両や、交差点で現れる車両を追い越しとして誤検出しないよう、背景のオプティカルフローを自車運動の代理値として使う。

連続フレーム間で背景特徴点の移動量を求め、track 区間内の中央値を `ego_motion` とする。`ego_motion < stop_flow` の場合、自転車が停止中とみなして候補から除外する。

このフィルタは既定で有効であり、`--no-stop-filter` で無効化できる。

### 2.4 軌跡フィルタ

bbox 下端中央の縦方向変化を、道路奥行き方向の移動として扱う。

```text
front: vertical_motion = y_start - y_end
rear : vertical_motion = y_end - y_start
lateral_motion = |cx_end - cx_start|
lateral_vertical_ratio = lateral_motion / max(|y_end - y_start|, 1e-6)
```

`vertical_motion < min_vertical_motion` の場合は、奥行き方向の動きが不足しているとして除外する。`lateral_vertical_ratio > max_lateral_vertical_ratio` の場合は、横方向主体の交差交通として除外する。

### 2.5 面積変化フィルタ

追い越し車は、撮影方向によって bbox 面積の変化方向が異なる。

```text
front: area_change_ratio = area_start / area_end
rear : area_change_ratio = area_end / area_start
```

`area_change_ratio < min_area_ratio` の場合は、接近・離脱に伴う面積変化が不足しているため除外する。

また、track 中盤だけ bbox が大きくなる横切り車両を除外するため、最大面積が中盤にあり、次の値が `max_mid_peak_ratio` を超える場合も除外する。

```text
mid_peak_ratio = peak_area / max(area_start, area_end)
```

### 2.6 道路 ROI

`--road-roi` を指定した場合、bbox 下端中央が台形の道路 ROI 内にある割合 `road_ratio` を計算する。`road_ratio < min_road_ratio` の track は除外する。

道路 ROI は GoPro の取り付け位置や画角に依存するため、既定では無効である。

## 3. 車種分類

車種は `大型`、`普通`、`軽` の3分類で扱う。

基本判定は次のとおり。

- YOLO が `truck` または `bus` と判定したフレームが一定以上続く場合は `大型`。
- YOLO が `car` と判定した場合は原則 `普通`。
- `car` の車体下部中央に黄色ナンバー形状が連続フレームで確認できた場合は `軽`。

画面端では GoPro の広角歪みにより車両が大きく見えやすいため、車種投票には bbox 面積を使わない。画面端の投票は弱く補正される。

大型車の黄色い注意表示などを軽と誤判定しないよう、`truck` / `bus` の強い判定を軽へ上書きする場合は、通常より高い黄色ナンバースコアを要求する。

## 4. 追い越し候補 CSV の時刻

`overtaking_events.csv` には、次の時刻が出力される。

| 列 | 意味 |
|---|---|
| `t_start_s` | 連結タイムライン上の候補開始時刻 |
| `peak_t_s` | bbox 面積が最大になる最接近目安時刻 |
| `t_end_s` | 連結タイムライン上の候補終了時刻 |
| `duration_s` | `t_end_s - t_start_s` |
| `source_video` | 候補が検出された元動画 |
| `local_t_start_s` | 元動画内の候補開始時刻 |
| `local_t_end_s` | 元動画内の候補終了時刻 |

複数動画をまとめて処理する場合、`t_*_s` は先頭動画からの累積時間、`local_*_s` は個別動画内の時刻になる。

## 5. GoPro と LiDAR の時刻同期

GoPro 動画内の GPMF から GPS UTC タイムスタンプ `GPSU` を読み取り、GPMF サンプルの presentation time を差し引いて、動画 `t=0` の UTC を推定する。

```text
GoPro開始Unix時刻 = GPSUのUnix時刻 - GPMFサンプル時刻
```

複数サンプルから推定値を作り、中央値付近の安定したクラスタを使う。

LiDAR 側は PCAP の最初のパケット時刻を開始 Unix 時刻とする。

GUI で使うオフセットは次で算出する。

```text
offset = GoPro開始Unix時刻 - PCAP開始Unix時刻
```

この実装では、レビュー済み CSV の VeloView 表示時刻は次で作る。

```text
VeloView時刻 = GoPro候補時刻 + offset
```

最終 Excel で GoPro GPS 速度を参照する際は、LiDAR 側の時刻を GoPro 動画内時刻へ戻すために、offset と複数動画の累積オフセットを差し引く。

```text
GoPro動画内時刻 = LiDAR相対時刻 - offset - 動画累積オフセット
```

## 6. LiDAR 点群の座標算出

PCAP から Velodyne の標準 1206-byte データパケットを読み込み、距離、方位角、垂直角から点群座標を算出する。

距離は Velodyne の raw distance を 0.002 倍して m に変換する。

```text
distance = distance_raw * 0.002
horizontal = distance * cos(vertical_angle)
X = horizontal * sin(azimuth)
Y = horizontal * cos(azimuth)
Z = distance * sin(vertical_angle)
```

対応する垂直角テーブルは LiDAR モデルごとに切り替える。

- VLP-16
- HDL-32E
- VLP-32C

GUI で矩形選択した範囲は、次の条件で点を抽出する。

```text
min_x <= X <= max_x
min_y <= Y <= max_y
laser指定がある場合は laser_id が指定集合に含まれる
```

出力 CSV では、選択点ごとに距離、方位角、仰角、`Points:0`、`Points:1`、`Points:2` を保存する。

```text
Distance = sqrt(X^2 + Y^2 + Z^2)
Azimuth = atan2(X, Y)
Elevation = atan2(Z, sqrt(X^2 + Y^2))
```

## 7. 最近接点 CSV の算出

`overtaking_XXX.csv` には、各フレームで選択された複数点が含まれる。`edit_csv.py` は各フレームから、XY 平面上で自転車に最も近い点を1点選ぶ。

```text
distance(m) = sqrt(X^2 + Y^2)
最近接点 = distance(m) が最小の点
```

出力される `overtaking_XXX_edit.csv` の列は次のとおり。

| 列 | 意味 |
|---|---|
| `unixtime` | LiDAR フレーム時刻。PCAP 内の相対秒、または元 CSV の時刻文字列。 |
| `X` | 最近接点の X 座標 |
| `Y` | 最近接点の Y 座標 |
| `Z` | 最近接点の Z 座標 |
| `distance(m)` | `sqrt(X^2 + Y^2)` |
| `speed(km/h)` | 前フレーム最近接点からの移動速度 |

速度は前フレームの最近接点との差分から算出する。

```text
speed(km/h) = sqrt((X_i - X_{i-1})^2 + (Y_i - Y_{i-1})^2)
              / (t_i - t_{i-1}) * 3600 / 1000
```

先頭フレームの速度は空欄になる。有効点がないフレームでは、時刻以外は空欄になる。

## 8. フェーズ分類

`edit.csv` の有効点から、次の4フェーズを決める。

| フェーズ | 定義 |
|---|---|
| フェーズ1 | 最初の有効点 |
| フェーズ2 | `abs(Y) < 3` になる前の点のうち、速度が一桁 km/h の最後の点。該当がなければ速度の絶対値が最小の点。 |
| フェーズ3 | フェーズ2以降で `abs(Y) < 3` になった最初の点。 |
| フェーズ4 | フェーズ3以降で `abs(Y) >= 3` になった最初の点。 |

3m は、車両が自転車に近接して通過している区間を Y 方向の位置で切り出すためのしきい値として使われる。`distance(m)` は `sqrt(X^2 + Y^2)` であり、フェーズ3・4の判定には使わない。

フェーズ2は、車両が近接区間に入る前の基準点として扱う。実装では、`abs(Y) < 3` になった時点で候補探索を打ち切り、それより前に速度値が入っている点だけをフェーズ2候補にする。候補の中に `0 <= abs(speed) < 10` の点がある場合は、その最後の点をフェーズ2にする。該当する一桁速度の点がない場合は、速度の絶対値が最も小さい点をフェーズ2にする。

フェーズ2は `distance_p2` と TTC の算出に使う。

```text
distance_p2 = abs(Y_phase2)
TTC = abs(Y_phase2) / (relative_speed / 3.6)
```

## 9. 分析指標の算出式

フェーズ分類後、次の指標を算出する。

### 9.1 離隔距離 LC

フェーズ3からフェーズ4までの区間を通過区間とする。そのうち `distance(m) >= 1.0` の点があればその点群を対象にし、なければ通過区間全体を対象にする。

```text
LC = abs(min(X))
```

`X` は横方向の座標として扱う。実装上は対象区間で最小の X を取り、その絶対値を離隔距離としている。

### 9.2 通過時間

```text
passing_time = t_phase4 - t_phase3
```

単位は秒。

### 9.3 距離 p2

```text
distance_p2 = abs(Y_phase2)
```

単位は m。

### 9.4 距離 p3

```text
distance_p3 = Y_phase4 - Y_phase3
```

単位は m。フェーズ3からフェーズ4までの Y 方向の移動量として扱う。

### 9.5 相対速度

```text
relative_speed(km/h) = distance_p3 / passing_time * 3.6
```

### 9.6 TTC

```text
TTC(s) = abs(Y_phase2) / (relative_speed / 3.6)
```

相対速度が 0 の場合、TTC は算出不能値になる。

## 10. GoPro GPS 速度の算出

GoPro の GPMF `GPS5` から GPS 速度を抽出し、`telemetry_gps.xlsx` を作成する。

主に使用する列は次のとおり。

- `cts`: GoPro 動画内時刻
- `GPS (2D speed) [m/s]`: 2D 速度

最終 Excel では、フェーズ3からフェーズ4までの GoPro 動画内時刻範囲に含まれる GPS 速度を平均し、km/h に変換する。

```text
bicycle_speed(km/h) = average(GPS 2D speed [m/s]) * 3.6
```

対象範囲に GPS 速度がない場合、自転車速度は空欄になる。

## 11. 自動車速度の算出

自動車速度は、自転車速度と LiDAR から求めた相対速度の和として出力する。

```text
vehicle_speed(km/h) = bicycle_speed(km/h) + relative_speed(km/h)
```

自転車速度または相対速度が欠けている場合、自動車速度は空欄になる。

## 12. 最終 Excel の出力項目

最終 Excel の主な列は次のとおり。

| 列 | 算出元 |
|---|---|
| 整理ID | GUI 入力。未入力時は候補 ID を使用。 |
| データ収集日 | GUI 入力、または出力フォルダ名から推定。 |
| 被験者名またはID | GUI 入力。 |
| 離隔距離（m） | `LC` |
| 自転車速度（km/h） | GoPro GPS 速度のフェーズ3から4の平均 |
| 自動車速度（km/h） | 自転車速度 + 相対速度 |
| 危険感 | GUI の危険感入力 |
| 動画ファイル名 | 候補の `source_video` |
| 動画のタイムスタンプ | `local_t_start_s` |
| LiDARデータファイル名 | PCAP ファイル名 |
| 日本時間 | PCAP 開始 Unix 時刻 + フェーズ3 LiDAR 時刻を JST 表示 |
| LiDARのタイムスタンプ | PCAP 開始 Unix 時刻 + フェーズ3 LiDAR 時刻 |
| 車種 | レビュー後の車種 |

## 13. 制約と確認ポイント

- GoPro の bbox 面積だけでは実距離を算出しない。距離・速度の最終算出には LiDAR 点群を使う。
- LiDAR の矩形選択がずれると、最近接点、LC、速度が直接ずれる。
- フェーズ3・4は `abs(Y) = 3m` のしきい値で決まるため、点群選択範囲と外れ値に注意する。
- GPS 速度は GoPro 側の記録に依存する。GPS が欠落している区間では自転車速度と自動車速度が空欄になる。
- 時刻同期の offset がずれると、候補時刻、LiDAR フレーム、GPS 速度参照範囲がすべてずれる。
- TTC は相対速度から計算した目安であり、相対速度が小さい場合や符号が不安定な場合は解釈に注意する。
